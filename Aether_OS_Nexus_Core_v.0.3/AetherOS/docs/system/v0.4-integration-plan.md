# Nexus Core v0.4 Integration Plan

This document converts the current architecture milestones into implementation work for **v0.4**, focusing on ABI unification, secure user-memory handling, and the first proxy-style hardware V-Node.

## Objectives

1. Unify syscall ABI to 64-bit `syscall` for all components using `common` syscall wrappers.
2. Enforce safe user-to-kernel memory transfer for pointer-carrying payloads (`Vec<u8>`, strings, slices).
3. Introduce IRQ proxy path for keyboard input (`IRQ1`) and route scan codes over IPC.
4. Close existing security gap in `SYS_LOG` by integrating `usercopy`.

## Current Status Snapshot (2026-03-28)

The original v0.4 workstreams are now largely in place:

- **ABI wrappers:** `common/src/syscall.rs` exposes x86_64 `syscall` wrappers (`syscall0..syscall6`) with a single 64-bit ABI contract.
- **Usercopy hardening:** `kernel/src/usercopy.rs` validates userspace ranges and `kernel/src/syscall.rs` routes `SYS_LOG` and IPC payload copies through those helpers.
- **Keyboard IRQ proxy:** `kernel/src/interrupts/keyboard.rs` captures IRQ1 scancodes from port `0x60` and forwards them over IPC to a registered channel.
- **Keyboard service:** `vnode/keyboard` registers IRQ1 with the kernel, ACKs interrupts via `SYS_IRQ_ACK`, forwards key events, and dispatches shell commands.

With these pieces connected, the remaining work is less about low-level plumbing and more about UX-level integration and stress validation.

## Constructor Priority Queue (Recommended Next Order)

To keep v0.4 focused and shippable, execute the next tasks in this strict order:

1. **P1 — End-to-end input routing contract (Keyboard → Focus Target).**
   - Define one canonical routing decision for each `KeyEvent` (`Shell`, `WebView`, or both).
   - Add an explicit `focus_owner` contract in docs so keyboard behavior is deterministic.
   - Exit criteria: `Enter` reaches shell command channel when shell has focus; text keys reach webview when webview has focus.

2. **P2 — Syscall ABI runtime verification hooks.**
   - Add an ABI smoke harness that calls `syscall0..syscall6` with known benign paths and validates argument positions.
   - Record expected/actual register-path behavior in test logs.
   - Exit criteria: one repeatable test confirms wrappers and kernel dispatch are aligned in QEMU boot.

3. **P3 — Usercopy fuzz/negative coverage.**
   - Add tests for out-of-range pointers, zero-length copies, and overflow-length arithmetic.
   - Ensure deterministic error codes are surfaced back to callers.
   - Exit criteria: invalid pointer cases reject safely without panic and are visible in kernel diagnostics.

4. **P4 — Stabilization pass for interactive loop.**
   - Validate keyboard IRQ throughput under sustained input.
   - Validate shell command dispatch + webview key injection under mixed workloads.
   - Exit criteria: no stuck IRQ ACK path, no channel starvation, no unbounded log spam.

> Rationale: this sequence closes user-visible behavior first (focus routing), then de-risks ABI and memory safety before throughput tuning.


## First Real V-Node Build Sequence (v0.4 execution kickoff)

Given the current state of the tree, both `logger` and `init-service` already exist as runnable V-Node binaries.
For **execution priority**, treat this as the first concrete rollout order:

1. **Logger as the first production anchor V-Node.**
   - Keep channel `9` as the canonical client ingress for `LoggerRequest`.
   - Validate request/response serialization under sustained `LogLevel::{Info,Warn,Error}` traffic.
   - Add monotonic timestamp decoration (from `SYS_TIME`) to each log line before forwarding to kernel log sink.

2. **Init-service as the second anchor V-Node.**
   - Expand bootstrap set beyond `echo` to include `logger`, `keyboard`, and `shell` once readiness contracts are finalized.
   - Enforce a strict service state machine: `Starting -> Ready -> Running -> Stopped`.
   - Persist an in-memory boot report for post-boot diagnostics over `InitRequest::ServiceStatus`.

3. **Integration gate between both.**
   - Route init lifecycle events through logger (`service started`, `service restart`, `bootstrap completed`).
   - Define one structured log schema for lifecycle events to support parsing in future observability tooling.

> Decision: **Start with Logger first, then Init-service orchestration hardening.**

## Workstream A — Syscall ABI Unification (Critical)

### Scope

- `common/src/syscall.rs`
- `kernel/src/syscall.rs`
- callsites using legacy assumptions from `int 0x80`

### Tasks

- Replace legacy interrupt-based userspace entry with 64-bit `syscall` convention.
- Verify register mapping consistency between userspace wrapper and kernel dispatcher.
- Add explicit ABI notes for enum/message payload passing used by IPC structures.
- Validate round-trip for representative IPC enums, including socket payloads.

### Definition of Done

- No remaining `int 0x80` syscall path in active runtime path.
- Syscall wrappers and kernel dispatcher agree on argument/return registers.
- Basic IPC flow passes for socket operations using new ABI path.

## Workstream B — Usercopy Integration for Data Safety

### Scope

- `kernel/src/syscall.rs`
- `kernel/src/usercopy.rs`
- syscall handlers that dereference user pointers

### Tasks

- Route `SocketRequest::Send` payload copy through `usercopy` before forwarding.
- Apply same pattern to `SYS_LOG` for string/byte input.
- Add defensive bounds checks and explicit error returns on invalid user memory.
- Normalize error mapping to user-visible errno-compatible codes.

### Definition of Done

- No direct unsafe dereference of user pointers in affected syscall handlers.
- Invalid userspace pointers fail gracefully without kernel crash or data leak.
- Kernel logs indicate rejected usercopy attempts with deterministic error code.

## Workstream C — Keyboard Proxy V-Node (IRQ1)

### Scope

- `kernel/src/interrupts/mod.rs`
- `vnode/keyboard/*`
- shared IPC contract used by shell/input consumers

### Tasks

- In IRQ1 handler, read scan code from port `0x60` and emit minimal IPC event.
- Keep kernel role limited to interrupt capture + delivery signaling.
- Grant `Capability::IrqAck` to keyboard service and move policy there.
- Implement key repeat/combo logic inside `keyboard` V-Node.
- Publish processed key events for shell consumption.

### Definition of Done

- Hardware keyboard IRQ path reaches keyboard V-Node through IPC.
- Kernel does not implement keymap/policy logic.
- Shell receives functional key events from keyboard service.

## Workstream D — Validation Matrix

- Syscall smoke test (open/log/ipc/send/recv).
- Socket IPC payload integrity test (`Send`/`Recv`).
- Keyboard IRQ-to-shell path test.
- Negative tests for invalid user pointers and oversized buffers.

## Delivery Sequence

1. ABI unification.
2. usercopy hardening for socket/log paths.
3. IRQ1 kernel patch.
4. Keyboard V-Node implementation.
5. Integrated validation pass.

### Suggested v0.4.x / v0.5 Follow-up Sequence ("Interactive Nexus")

1. **Display-compositor integration:** render UI protocol updates sourced from HTML/CSS/layout output.
2. **Keyboard-to-webview focus routing:** map key events to focused view/component instead of global shell-only routing.
3. **Backpressure + rate limiting:** guard against log/IPC floods from layout-heavy scenes.
4. **End-to-end regression suite:** scripted QEMU test that verifies keyboard IRQ -> shell command -> UI refresh loop.

### Decision Point: Compositor vs Network Stack (Current Recommendation)

If only one track can be staffed immediately, prioritize the **graphical compositor path first** and keep network work as a constrained parallel spike.

1. **Why compositor first**
   - It unlocks visible end-to-end proof that current IPC contracts (`ui_protocol`, keyboard focus routing, logger diagnostics) behave correctly under real user interaction.
   - It stress-tests the existing scheduler/IPC loop with deterministic local inputs before adding nondeterministic network timing and failure modes.
   - It shortens feedback cycles: UI regressions are immediately observable in QEMU, while network regressions often need additional harnesses and external fixtures.

2. **Network scope during compositor focus**
   - Keep socket work in a thin vertical slice only: connect/send/recv/close with strict timeouts and bounded buffers.
   - Defer advanced features (retry orchestration, protocol multiplexing, mail sync acceleration) until compositor + input + logger telemetry are stable.

3. **Readiness gate to switch priority to network-heavy work**
   - Compositor can render and update at least one focused shell/webview workflow without starvation.
   - Logger channel `9` shows structured diagnostics for UI/input flows without flood-induced degradation.
   - Keyboard focus routing remains deterministic across repeated interaction tests.

## Risks and Mitigations

- **ABI mismatch risk:** enforce single source-of-truth register map in comments + tests.
- **Pointer safety regressions:** gate all pointer-based syscalls through usercopy helpers.
- **IRQ event loss:** keep IRQ handler minimal and non-blocking; defer logic to V-Node.

## Expected Outcome

After these steps, Nexus Core transitions from static interface design to a functional service mesh where shell input, AI processing, and socket networking can operate end-to-end over the same IPC-first model.
